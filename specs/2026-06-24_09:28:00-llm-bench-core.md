## 1. Executive Summary

`llm-bench` is a Python (asyncio) command-line tool that benchmarks the performance and optionally the output quality of Large Language Model endpoints exposed through an OpenAI-compatible API (`/v1/chat/completions` in streaming). It targets both self-hosted servers (vLLM, TGI, SGLang, llama.cpp) and hosted APIs.

The tool produces time-deterministic measurements: each run holds load for a fixed wall-clock duration per concurrency level rather than for a fixed request count. It measures latency (TTFT, TPOT, ITL, E2E), throughput (per-user and system token rates, RPS), reliability (success, 429, timeout, error rates), goodput (SLO-aware throughput), and cost. When prompts carry an expected output, an asynchronous evaluation pipeline scores quality (embedding cosine similarity by default, optional LLM-as-judge) without perturbing the load test timing.

Results are persisted as per-request JSONL, rolled up to Parquet, queryable through DuckDB, summarized in a rich terminal table, and rendered as a self-contained static HTML report (Plotly). The audience is the operator running capacity and model-comparison benchmarks (a performance engineer or architect).

This specification consolidates the project's design decisions and covers the complete tool across five implementation phases.

## 3. Scope

### 3.1 In Scope
- Closed-loop concurrency sweep, duration-based per level (warmup, steady, cooldown phases).
- Open-loop Poisson arrival mode at fixed request rates, with a `max_outstanding` guard.
- Full latency, throughput, reliability, goodput, and cost metrics with mean/min/max/std/p50/p90/p95/p99 (p99.9 when samples suffice).
- Configuration driven by `config.yaml` with `$ENV:VAR` interpolation; a model registry; SLO profiles.
- Built-in prompt library plus user override, with deterministic mock tools (web search, pptx) and multimodal image prompts.
- Anti-cache controls (seeded random prompt selection, unique per-request prefix, deterministic output length).
- Asynchronous quality evaluation (embedding cosine default, optional LLM-as-judge), decoupled by queue, joined on `request_id`.
- Persistence: per-request JSONL, Parquet rollup, summary JSON, resolved config and environment snapshot.
- Reporting: rich terminal summary, self-contained static HTML report, and a DuckDB-backed `analyze` command.
- Structured logging and minimal OpenTelemetry tracing (JSONL collector) for internal LLM calls.

### 3.2 Out of Scope (Non-Goals)
- Native non-OpenAI protocols (Anthropic, Vertex, Bedrock native SDKs). Only the OpenAI-compatible API is supported; a compatibility gateway is expected when needed.
- Distributed multi-machine load generation. A single asyncio process on one machine is the design target.
- A live web dashboard server (Grafana, default Streamlit). Output is the static HTML report plus terminal; an optional Streamlit explorer over Parquet is a deferred extension (backlog).
- Academic multi-dataset leaderboard evaluation (MMLU and similar). Quality evaluation is limited to comparing actual output against an expected output per prompt.

## 4. User Personas & Actors

- **Operator** (primary, human): a performance engineer or architect who configures the model registry, launches benchmark runs, reviews terminal and HTML reports, and runs ad-hoc DuckDB analysis.
- **Eval worker pool** (system actor): a decoupled set of asynchronous workers that consume the evaluation queue and score quality (embedding or judge) independently of the load generator.

## 5. Usage Scenarios

### SC-001: Closed-loop concurrency sweep
**Actor:** Operator
**Preconditions:** A valid `config.yaml` is present; the target model endpoint is reachable; a prompt set (built-in or supplied) is loaded.
**Flow:**
1. The operator runs `llm-bench run --config config.yaml --model <name> --concurrency 1,2,5,10 --duration 180s`.
2. The system validates config, resolves `$ENV:` references, and performs a pre-flight verification request.
3. For each concurrency level sequentially, the system runs warmup (unmeasured), steady (measured), and cooldown phases, streaming requests with `stream_options.include_usage=true`.
4. The system measures TTFT, time-to-second-token, TPOT, ITL, E2E and reads token usage per request, tagging each request with its phase, category, and length buckets.
5. The system writes one JSONL record per request during the run and prints a rich terminal summary per level.
**Postconditions:** Raw JSONL, Parquet rollup, summary JSON, and resolved config and environment snapshots are written; a terminal summary is displayed.
**Exceptions:**
- EXC-001a: Pre-flight verification fails (connection refused, DNS, 401, 5xx) → abort with non-zero exit, descriptive message, no run data.
- EXC-001b: A request returns HTTP 429 → record as `rate_limited`, no retry, continue; surface and flag the 429 rate if it exceeds 1%.
- EXC-001c: A request exceeds the timeout or the stream breaks → record as `timeout` or `malformed_stream` failure, excluded from latency distributions.
- EXC-001d: A steady window collects fewer than `min_samples` → emit a warning, continue.
- EXC-001e: The operator interrupts the run (SIGINT) → flush buffered data, drain the eval queue, mark the run `incomplete`.

### SC-002: Open-loop Poisson rate sweep
**Actor:** Operator
**Preconditions:** Same as SC-001; `mode: open` and one or more `request_rates`.
**Flow:**
1. The operator runs `llm-bench run --model <name> --mode open --request-rate 5,10,20 --duration 180s`.
2. The system generates request arrivals at each configured rate with Poisson inter-arrivals (`burstiness`), independent of completion times.
3. The system enforces the `max_outstanding` guard, measures the same metric set, and computes goodput against the active SLO profile.
**Postconditions:** Raw data and summary include `mode: open`, `level_or_rate` set to the rate, and goodput metrics.
**Exceptions:**
- EXC-002a: `request_rate <= 0` → reject with non-zero exit.
- EXC-002b: Offered rate exceeds server capacity and outstanding requests reach `max_outstanding` → pause new arrivals, warn, count the guard events.
**Cross-scenario notes:** Open-loop arrivals avoid the coordinated omission bias that understates tail latency in the closed-loop SC-001 under saturation.

### SC-003: Generate the HTML report
**Actor:** Operator
**Preconditions:** A run directory exists with valid `raw.jsonl` and `summary.json`.
**Flow:**
1. The operator runs `llm-bench report runs/<id>/`.
2. The system aggregates the raw data through DuckDB and renders a self-contained static HTML report with the standard chart set.
**Postconditions:** `report.html` is written, fully offline, sharable as a single file.
**Exceptions:**
- EXC-003a: Raw data is missing → abort with a descriptive error naming the expected path.
- EXC-003b: Raw data is unreadable or corrupt → abort with a parse error, no HTML written.

### SC-004: Run with quality evaluation
**Actor:** Operator and Eval worker pool
**Preconditions:** Prompts carry `expected_output`; `evaluation` config is valid (a `threshold` is mandatory for embedding).
**Flow:**
1. The operator runs `llm-bench run ... --eval-method embedding` (or `judge`).
2. On each completed request with an expected output, the load generator enqueues an eval record non-blocking.
3. The eval worker pool consumes the queue with its own rate limiter, computes embedding cosine similarity (or an LLM-as-judge verdict), and writes results keyed by `request_id`.
4. When the load test ends, the system publishes performance metrics immediately and continues draining the eval queue (backfill); remaining items past a global timeout are marked `eval_skipped`.
5. The system joins eval scores to performance records on `request_id` and reports coverage.
**Postconditions:** Joined records carry quality scores; coverage (`judged/total`) is reported.
**Exceptions:**
- EXC-004a: `threshold` missing while embedding eval is active → abort at config validation.
- EXC-004b: Judge or embedding endpoint unreachable → mark records `eval_skipped`, coverage below 1.0, performance still valid.
- EXC-004c: Eval queue saturates → drop-with-counter or spill, never block the load generator.

### SC-005: Tooling and multimodal prompts
**Actor:** Operator
**Preconditions:** The model declares `supports_tools` and/or `supports_vision`; deterministic mock tools are defined.
**Flow:**
1. The operator runs a bench whose prompt set includes tool-use prompts (mock web search, mock pptx) and image prompts.
2. The system exposes deterministic mock tools via the OpenAI `tools` field, returning fixed results so synthesis and tool-calling are measured without network variance.
3. The system tags each request with its category and measures the standard metrics.
**Postconditions:** Metrics segmented by category; tool results are deterministic.
**Exceptions:**
- EXC-005a: A prompt requires a capability the model does not declare → skip the prompt with a warning, continue on compatible prompts.

### SC-006: Ad-hoc analysis via DuckDB
**Actor:** Operator
**Preconditions:** A run produced `raw.jsonl` or `rollup.parquet`.
**Flow:**
1. The operator runs `llm-bench analyze runs/<id>/raw.jsonl --sql "<query>"`.
2. The system executes the query through DuckDB directly on the file and prints the result.
**Postconditions:** Query results displayed; no data mutated.
**Exceptions:**
- EXC-006a: The data file is absent → abort with a descriptive error.
- EXC-006b: The SQL is invalid → return the DuckDB parser error with a non-zero exit.

### SC-007: Configure and validate the model registry
**Actor:** Operator
**Preconditions:** A `config.yaml` exists.
**Flow:**
1. The operator runs `llm-bench run ... --dry-run` (validate only) or any run.
2. The system loads and validates the config (Pydantic), resolves `$ENV:VAR` references, and persists a resolved snapshot with secrets redacted.
**Postconditions:** Config validated; resolved snapshot written; secrets never persisted in clear.
**Exceptions:**
- EXC-007a: A referenced env var is unset → abort, naming the variable.
- EXC-007b: The YAML is malformed → abort with a parse error.
- EXC-007c: Embedding eval lacks `threshold` → abort at config validation.

## 6. Functional Requirements

EARS notation is mandatory. Forbidden modals (should/may/could/might/would) are not used. Patterns: U (ubiquitous), E (event), S (state), O (optional/conditional), UB (unwanted behavior), X (free-form escape).

### Configuration

#### FR-001 [EARS-E]: Load and resolve config
> WHEN a config file is loaded THE system SHALL validate it against the schema and resolve every `$ENV:VAR` reference to its environment value.
- **Inputs:** path to `config.yaml`; process environment.
- **Outputs:** a validated, resolved config object.
- **Business Rules:** validation uses a typed schema (Pydantic); no `$ENV:` token remains after resolution.
- **Priority:** Must-have

#### FR-002 [EARS-O]: Missing env var aborts
> IF a referenced environment variable is not set THEN THE system SHALL abort with a non-zero exit code and an error naming the variable.
- **Inputs:** an unresolved `$ENV:NAME`.
- **Outputs:** stderr `environment variable not set: NAME`; exit code 2.
- **Business Rules:** no run data is written.
- **Priority:** Must-have

#### FR-003 [EARS-O]: Embedding eval requires threshold
> IF the active evaluation method is embedding AND `evaluation.embedding.threshold` is absent THEN THE system SHALL abort with an explanatory error.
- **Inputs:** evaluation config.
- **Outputs:** stderr `embedding evaluation requires evaluation.embedding.threshold`.
- **Business Rules:** validated at config stage before contacting any endpoint.
- **Priority:** Must-have

#### FR-004 [EARS-U]: Model registry
> THE system SHALL support a model registry where each entry carries `base_url`, `model`, optional `api_key`, optional `tokenizer`, `supports_vision`, `supports_tools`, and optional `price_input`/`price_output`.
- **Inputs:** registry section of config.
- **Outputs:** typed registry entries.
- **Business Rules:** `api_key` is optional; pricing is optional.
- **Priority:** Must-have

### Closed-loop load generation

#### FR-005 [EARS-E]: Sequential concurrency levels
> WHEN a closed-loop run starts THE system SHALL execute each configured concurrency level sequentially for the configured duration.
- **Inputs:** `concurrency_levels`, `duration`.
- **Outputs:** measured requests tagged by level.
- **Business Rules:** levels do not overlap; each level value MUST be >= 1.
- **Priority:** Must-have

#### FR-006 [EARS-U]: Phase tagging
> THE system SHALL run warmup, steady, and cooldown phases per level and tag each request with its phase, reporting metrics over steady requests only.
- **Inputs:** `warmup`, `cooldown`, `duration`.
- **Outputs:** `phase` field per record.
- **Business Rules:** `warmup + cooldown` MUST be less than `duration`.
- **Priority:** Must-have

#### FR-007 [EARS-U]: Streaming with usage
> THE system SHALL stream requests against the OpenAI-compatible chat completions endpoint with `stream=true` and `stream_options.include_usage=true`.
- **Inputs:** request payload.
- **Outputs:** streamed SSE chunks plus a final usage object.
- **Business Rules:** if the final usage chunk is absent, the system falls back to counting deltas and flags `usage_incomplete`.
- **Priority:** Must-have

#### FR-008 [EARS-O]: Minimum samples warning
> IF a concurrency level collects fewer than `min_samples` steady requests THEN THE system SHALL emit a warning and set `min_samples_met` false for that level.
- **Inputs:** steady sample count, `min_samples`.
- **Outputs:** warning `steady samples N < min_samples M`.
- **Business Rules:** non-fatal; the run continues.
- **Priority:** Must-have

### Pre-flight and reliability

#### FR-009 [EARS-E]: Pre-flight verification
> WHEN a run is initiated THE system SHALL issue a single pre-flight verification request before any measured load.
- **Inputs:** model endpoint.
- **Outputs:** a verification request not recorded in `raw.jsonl`.
- **Priority:** Must-have

#### FR-010 [EARS-O]: Pre-flight failure aborts
> IF the pre-flight verification request fails THEN THE system SHALL abort with a non-zero exit code and a descriptive message and SHALL write no measured run data.
- **Inputs:** pre-flight response or error.
- **Outputs:** stderr `pre-flight verification failed` plus the cause.
- **Priority:** Must-have

#### FR-011 [EARS-O]: 429 recorded without retry
> IF a request returns HTTP 429 THEN THE system SHALL record it as `rate_limited` with `retry_count=0` and continue the run.
- **Business Rules:** `retries` defaults to 0; no silent retry.
- **Priority:** Must-have

#### FR-012 [EARS-U]: Surface 429 rate
> THE system SHALL report the 429 rate per level and flag it when it exceeds 1 percent.
- **Outputs:** `rate_limited_rate`, `rate_limited_flagged`.
- **Business Rules:** the flag threshold is strictly greater than 1 percent.
- **Priority:** Must-have

#### FR-013 [EARS-O]: Timeout and stream failures
> IF a request exceeds the timeout or its stream breaks or yields unparseable data THEN THE system SHALL record it as a `timeout` or `malformed_stream` failure.
- **Outputs:** `outcome`, `error`.
- **Priority:** Must-have

#### FR-014 [EARS-UB]: Exclude failures from distributions
> THE system SHALL NOT include failed or timed-out requests in latency or throughput distributions.
- **Business Rules:** failures are counted separately as reliability rates.
- **Priority:** Must-have

#### FR-015 [EARS-E]: Graceful interruption
> WHEN the operator sends SIGINT THE system SHALL flush buffered raw data, drain the evaluation queue, and mark the run `incomplete`.
- **Outputs:** valid partial `raw.jsonl`; `summary.json` with `status: incomplete`; exit code 130.
- **Priority:** Must-have

### Open-loop load generation

#### FR-016 [EARS-E]: Poisson arrivals
> WHEN an open-loop run starts THE system SHALL generate request arrivals at the configured rate with Poisson inter-arrivals controlled by `burstiness`, independent of completion times.
- **Business Rules:** `request_rate` MUST be greater than 0; arrival schedule is seeded for reproducibility.
- **Priority:** Must-have

#### FR-017 [EARS-S]: Outstanding guard
> WHILE outstanding requests reach `max_outstanding` THE system SHALL pause launching new arrivals until capacity frees and SHALL count the guard events.
- **Outputs:** `max_outstanding_events`; warning `max_outstanding reached`.
- **Priority:** Must-have

### Metrics

#### FR-018 [EARS-U]: TTFT
> THE system SHALL compute TTFT as the time from request send to the first content-bearing chunk, ignoring an initial role-only chunk.
- **Priority:** Must-have

#### FR-019 [EARS-U]: TPOT
> THE system SHALL compute TPOT as `(E2E - TTFT) / (output_tokens - 1)`, request-weighted.
- **Business Rules:** when `output_tokens` equals 1, TPOT is null (no division).
- **Priority:** Must-have

#### FR-020 [EARS-U]: ITL
> THE system SHALL compute ITL as the token-weighted distribution of inter-token gaps, excluding the TTFT gap.
- **Priority:** Must-have

#### FR-021 [EARS-U]: Additional latencies
> THE system SHALL compute E2E latency, time-to-second-token, and normalized latency (`E2E / output_tokens`).
- **Priority:** Must-have

#### FR-022 [EARS-U]: Throughput metrics
> THE system SHALL report per-user output tokens per second, system output tokens per second, requests per second, and total tokens per second.
- **Priority:** Must-have

#### FR-023 [EARS-U]: Statistical aggregation
> THE system SHALL report latency metrics as mean, min, max, std, p50, p90, p95, and p99.
- **Priority:** Must-have

#### FR-024 [EARS-O]: p99.9 conditional
> IF a level has at least the documented sample threshold THEN THE system SHALL also report p99.9, otherwise it SHALL omit it.
- **Priority:** Should-have

#### FR-025 [EARS-U]: Token source
> THE system SHALL derive `output_tokens` from the server `usage` field.
- **Priority:** Must-have

#### FR-026 [EARS-UB]: No max_tokens accounting
> THE system SHALL NOT compute throughput or token counts from `max_tokens`.
- **Priority:** Must-have

#### FR-027 [EARS-U]: Cached and reasoning tokens
> THE system SHALL track `cached_tokens` and `reasoning_tokens` from the usage details as separate fields.
- **Priority:** Must-have

#### FR-028 [EARS-O]: Cache-bias warning
> IF cache busting is enabled AND a response reports `cached_tokens` greater than 0 THEN THE system SHALL emit a cache-bias warning and count the violation.
- **Priority:** Must-have

### Goodput

#### FR-029 [EARS-U]: Goodput
> THE system SHALL compute goodput as the count and rate of requests meeting all SLO thresholds of the active profile, plus the attainment fraction.
- **Priority:** Must-have

#### FR-030 [EARS-U]: SLO profiles
> THE system SHALL provide interactive and relaxed SLO profiles, overridable in config and via CLI.
- **Priority:** Must-have

### Cost

#### FR-031 [EARS-O]: Cost computation
> IF a model defines `price_input` and `price_output` THEN THE system SHALL compute per-request cost, cost per 1000 requests, and total cost.
- **Priority:** Should-have

#### FR-032 [EARS-O]: Silent cost omission
> IF pricing is absent THEN THE system SHALL omit cost metrics without warning or error.
- **Priority:** Should-have

### Prompts and anti-cache

#### FR-033 [EARS-U]: Seeded prompt selection
> THE system SHALL select prompts randomly per request using a seeded random generator so selection is reproducible.
- **Priority:** Must-have

#### FR-034 [EARS-O]: Unique prefix
> IF cache busting is enabled THEN THE system SHALL prepend a unique prefix to each request prompt.
- **Priority:** Must-have

#### FR-035 [EARS-O]: Deterministic output length
> IF `ignore_eos` and `max_tokens` are set THEN THE system SHALL request a deterministic output length.
- **Priority:** Should-have

#### FR-036 [EARS-U]: Prompt library
> THE system SHALL ship a built-in prompt library and SHALL load an external prompt set when `--prompts` is given.
- **Business Rules:** an empty prompt set aborts with `no prompts loaded`.
- **Priority:** Must-have

#### FR-037 [EARS-U]: Request tagging
> THE system SHALL tag each request with its prompt category and its input and output length buckets.
- **Priority:** Must-have

### Tooling and multimodal

#### FR-038 [EARS-E]: Deterministic mock tools
> WHEN a tool-use or multimodal prompt is executed THE system SHALL expose deterministic mock tools via the OpenAI `tools` field returning fixed results, and SHALL pass multimodal image content unchanged.
- **Priority:** Should-have

#### FR-039 [EARS-O]: Capability mismatch skip
> IF a prompt requires a capability the model does not declare THEN THE system SHALL skip that prompt with a warning, count it, and continue.
- **Priority:** Must-have

### Evaluation (asynchronous)

#### FR-040 [EARS-E]: Non-blocking enqueue
> WHEN a request with an expected output completes THE system SHALL enqueue an evaluation record without blocking the load generator.
- **Priority:** Must-have

#### FR-041 [EARS-UB]: No backpressure on load gen
> THE system SHALL NOT block the load generator on evaluation backpressure; it SHALL drop-with-counter or spill instead.
- **Priority:** Must-have

#### FR-042 [EARS-U]: Separate eval pool
> THE system SHALL run evaluation workers in a separate pool with its own rate limiter on the judge or embedding provider.
- **Priority:** Must-have

#### FR-043 [EARS-U]: Embedding default
> THE system SHALL compute embedding cosine similarity by default and apply the configured threshold to produce a pass or fail.
- **Business Rules:** the threshold is inclusive (greater than or equal). A prompt with empty expected output is skipped and excluded from coverage.
- **Priority:** Must-have

#### FR-044 [EARS-O]: Judge method
> IF the evaluation method is judge THEN THE system SHALL grade the output with the configured judge model using a binary or three-level rubric.
- **Business Rules:** no numeric 1 to 10 scale; the system warns if the judge model family matches the model under test (SUT).
- **Priority:** Should-have

#### FR-045 [EARS-E]: Publish then backfill
> WHEN the load test ends THE system SHALL publish performance metrics immediately and continue draining the evaluation queue, backfilling scores.
- **Priority:** Must-have

#### FR-046 [EARS-O]: Eval timeout coverage
> IF the evaluation global timeout elapses THEN THE system SHALL mark remaining records `eval_skipped` and report coverage.
- **Priority:** Must-have

#### FR-047 [EARS-U]: Join on request_id
> THE system SHALL join evaluation scores to performance records on `request_id`.
- **Priority:** Must-have

### Storage

#### FR-048 [EARS-U]: Per-request JSONL
> THE system SHALL write exactly one JSONL record per measured request with a unique `request_id`.
- **Business Rules:** a write failure aborts with `cannot write run data`.
- **Priority:** Must-have

#### FR-049 [EARS-U]: ITL storage
> THE system SHALL store an ITL summary per request by default and the full ITL list only when `--raw-itl` is given.
- **Priority:** Must-have

#### FR-050 [EARS-E]: Rollup at end
> WHEN a run completes THE system SHALL roll up the raw JSONL to Parquet and write a summary JSON.
- **Priority:** Must-have

#### FR-051 [EARS-U]: Config and env snapshot
> THE system SHALL persist a resolved config snapshot and an environment snapshot per run.
- **Business Rules:** secrets are redacted in the resolved config.
- **Priority:** Must-have

#### FR-052 [EARS-U]: Analyze command
> THE system SHALL provide an `analyze` command that runs a DuckDB query directly on a run's raw or Parquet data.
- **Business Rules:** a missing file or invalid SQL aborts with a descriptive error.
- **Priority:** Should-have

### Reporting

#### FR-053 [EARS-U]: Terminal summary
> THE system SHALL print a rich terminal summary with percentile tables and reliability rates per level.
- **Priority:** Must-have

#### FR-054 [EARS-E]: HTML report
> WHEN report generation is requested THE system SHALL produce a self-contained static HTML report containing the standard chart set.
- **Business Rules:** the report is fully offline with no external network references.
- **Priority:** Must-have

#### FR-055 [EARS-O]: Report data guard
> IF raw data for a run is missing or unreadable THEN THE system SHALL abort report generation with a descriptive error.
- **Priority:** Must-have

### Observability and integrity

#### FR-056 [EARS-U]: Logs and traces
> THE system SHALL emit structured logs and OpenTelemetry traces (JSONL collector) for internal LLM calls recording model, token counts, duration, and cost.
- **Priority:** Should-have

#### FR-057 [EARS-UB]: No secret or content leakage
> THE system SHALL NOT record prompts, responses, credentials, or API keys in traces, logs, or the resolved config snapshot.
- **Priority:** Must-have

#### FR-058 [EARS-U]: Monotonic clock
> THE system SHALL use a monotonic clock for all duration measurements.
- **Priority:** Must-have

#### FR-059 [EARS-O]: Event-loop lag warning
> IF event-loop lag exceeds the configured threshold during a level THEN THE system SHALL emit a warning of possible client-side saturation bias.
- **Priority:** Should-have

## 7. Non-Functional Requirements

### 7.1 Performance
- The load generator MUST sustain up to 1000 concurrent virtual users on a single machine; the httpx connection pool MUST be sized to at least peak concurrency.
- Client-side per-chunk processing overhead MUST remain negligible relative to inter-token gaps; the tool MUST monitor event-loop lag (FR-059) and warn when client saturation could bias measurements.
- All durations MUST use `time.monotonic()` (FR-058).

### 7.2 Security
- The only secrets are API keys supplied through `$ENV:` interpolation; no secret is persisted in clear (FR-057, FR-051).
- There is no web authentication surface (local CLI). Security testing focuses on secret non-leakage in traces, logs, and snapshots, and on correct env resolution.

### 7.3 Usability
- The tool ships a built-in prompt library so a benchmark is runnable immediately after install (FR-036).
- The terminal summary surfaces reliability (429, timeout, error rates) prominently alongside latency.
- Warnings (min samples, cache bias, capability mismatch, event-loop lag) are explicit and actionable.

### 7.4 Reliability
- Failed and timed-out requests are isolated from latency distributions (FR-014) and reported as separate rates.
- SIGINT performs a clean flush and marks the run incomplete (FR-015); partial data remains usable.

### 7.5 Observability
- **Collector:** JSONL file (default), path `runs/<id>/traces.jsonl`.
- **What to trace:** internal LLM calls (model under test pre-flight, judge, embedding) at INFO with model, token counts, duration, cost; errors at ERROR; warnings at WARNING. Per-request benchmark measurements remain in the dedicated `raw.jsonl`, separate from traces.
- **LLM tracing:** model, token counts, duration, cost only.
- **Sensitive data exclusion:** prompts, responses, credentials, API keys are never written to traces or logs (FR-057).
- Logs are structured (JSON) with at least `timestamp`, `level`, `event`.

### 7.6 Deployment
- Distribution: local Python project managed by `uv`, installed editable, exposing a console entry point `llm-bench`. No PyPI publication, no binary packaging.
- License: MIT.
- No cloud infrastructure, no service deployment (CLI tool).

### 7.7 Scalability
- Single-process asyncio on one machine is the design boundary (distributed generation is a non-goal). Concurrency scales vertically to 1000+ virtual users.

### Code quality (Maintainability)
- Python 3.12+, managed with `uv`, linted with Ruff, type-checked with mypy, tested with pytest. E2E tests in `tests/` are the acceptance contract.

## 8. Data Model

Primary persisted entity is the per-request record (`raw.jsonl`):

| Field | Type | Notes |
|-------|------|-------|
| run_id | str | run identifier |
| model | str | model under test |
| mode | str | `closed` or `open` |
| level_or_rate | number | concurrency level or arrival rate |
| phase | str | `warmup`, `steady`, `cooldown` |
| seed | int | master seed |
| request_id | str (uuid) | unique join key |
| prompt_id | str | selected prompt |
| category | str | `coding`, `synthesis`, `tool-use`, `vision`, `general` |
| isl_bucket / osl_bucket | str | input/output length buckets |
| prompt_tokens / output_tokens | int | from server usage |
| cached_tokens / reasoning_tokens | int | from usage details |
| usage_incomplete | bool | true when final usage missing |
| t_start | float | monotonic start |
| ttft / tt2t / e2e / tpot | float\|null | latencies (ms or s) |
| itl_summary | object | `{mean,p50,p95,p99,max}` |
| itl_list | array | present only with `--raw-itl` |
| outcome | str | `success`, `rate_limited`, `timeout`, `malformed_stream`, `connection_error` |
| status_code | int | HTTP status |
| retry_count | int | always 0 by default |
| error | str\|null | failure detail |
| cost_usd | float\|null | present only when pricing defined |
| sim_score | float\|null | embedding cosine |
| quality_pass | bool\|null | threshold applied |

## 12. End-to-End Test Suite

> E2E tests are the primary contract. Every endpoint under test is replaced by a local fake OpenAI-compatible streaming server (`FakeSUT`) plus fake judge/embedding endpoints (`FakeEval`), all deterministic and offline. Every "Then" asserts an observable: JSONL/Parquet/summary field values, computed metrics within tolerance, exit codes, stderr/log strings, HTML content, or DuckDB query results.

### Common harness
- **FakeSUT**: in-process server at `http://127.0.0.1:{port}/v1`, driven by a per-test behavior script controlling role-only first chunk, ordered content deltas with per-delta sleep, the final `usage` object, HTTP status (200/429/500), forced timeout, mid-stream TCP close, malformed SSE line, omission of the final usage chunk, and echo of received `tools`/multimodal content. SSE events are `data: {json}\n\n`, terminated by `data: [DONE]\n\n`.
- **FakeEval**: fake `/v1/embeddings` (fixed vectors so cosine is hand-computable) and fake judge `/v1/chat/completions` (fixed JSON verdict); supports "down" (connection refused) and "slow" modes.
- **Time control**: delays are real but tiny (ms); metric assertions use ±15% tolerance unless stated; production clock is `time.monotonic()` (durations asserted positive and ordered, not absolute).
- **CFG_BASE** default config:
```yaml
models:
  - name: sut
    base_url: http://127.0.0.1:${SUT_PORT}/v1
    model: fake/model
    api_key: $ENV:SUT_API_KEY
    supports_vision: false
    supports_tools: false
run:
  mode: closed
  duration: 2s
  warmup: 0.5s
  cooldown: 0.5s
  min_samples: 30
  concurrency_levels: [1, 2]
  max_tokens: 8
  ignore_eos: true
  temperature: 0.0
  cache_busting: true
  retries: 0
  timeout: 5s
  seed: 42
  slo_profile: interactive
```
Env preset: `SUT_API_KEY=sk-test`, `SUT_PORT` set by harness.

## 16. Glossary

| Term | Definition | Context |
|------|------------|---------|
| TTFT | Time To First Token: time from request send to first content chunk | Metrics |
| TPOT | Time Per Output Token: `(E2E - TTFT)/(output_tokens - 1)`, request-weighted | Metrics |
| ITL | Inter-Token Latency: token-weighted distribution of inter-token gaps, excluding TTFT | Metrics |
| E2E | End-to-end latency: send to last token | Metrics |
| Goodput | Throughput counting only requests meeting all SLO thresholds | Metrics |
| Closed-loop | Fixed number of virtual users, each sending the next request on completion | Load model |
| Open-loop | Fixed arrival rate (Poisson), independent of completion | Load model |
| Coordinated omission | Underestimation of tail latency in closed-loop under saturation | Load model |
| Warmup/steady/cooldown | Per-level phases; metrics computed over steady only | Methodology |
| Cache busting | Unique per-request prefix to avoid prefix-cache bias | Anti-cache |
| ISL / OSL | Input / Output Sequence Length buckets | Dimensions |
| SUT | System Under Test: the model endpoint being benchmarked | Benchmark |
| SLO profile | Named set of latency thresholds (interactive, relaxed) for goodput | Configuration |
| LLM-as-judge | Scoring output quality with a separate judge model | Evaluation |
| Embedding cosine | Semantic similarity of expected vs actual via embedding vectors | Evaluation |
| Coverage | Fraction of eligible requests actually evaluated (judged/total) | Evaluation |

## 17. Interview Decisions Log

- **DEC-001:** Language is Python (asyncio + httpx). **Rationale:** richest ecosystem for tokenizers, embeddings, judge, Plotly, DuckDB, and aligned with the user's Python tooling; concurrency is I/O-bound so asyncio suffices. **Alternatives:** Go (better raw concurrency, weaker eval/report ecosystem), Rust (fastest, slowest to build). **Round:** 1
- **DEC-002:** Distribution is local via uv editable install with a console entry point. **Rationale:** personal tool, simplest path. **Alternatives:** PyPI publish, pyinstaller binary. **Round:** 1
- **DEC-003:** Observability is structured logs + minimal OTel JSONL for internal LLM calls; benchmark measurements stay in their own JSONL. **Rationale:** the tool already produces measurement data; OTel on internal calls only, never prompts/responses. **Alternatives:** OTLP to Jaeger/Tempo, logs only. **Round:** 1
- **DEC-004:** Non-goals: native non-OpenAI protocols, distributed multi-machine generation, live web dashboard server. **Round:** 1
- **DEC-005:** Spec covers all five phases. **Rationale:** the user requires the full tool implemented autonomously in sequence. **Round:** 1
- **DEC-006:** Both closed-loop and open-loop modes are delivered (closed default). **Rationale:** closed matches the N-users mental model; open gives honest tail latencies (coordinated omission). **Round:** 2
- **DEC-007:** Failure policy: pre-flight then abort; 429 continue+flag; SIGINT clean flush+incomplete; eval backfill with global timeout. **Round:** 2c
- **DEC-008:** Ship a built-in prompt library overridable via `--prompts`; capability mismatch skips with a warning; runs are multi-category tagged per request. **Round:** 3
- **DEC-009:** License MIT; Python 3.12+; sustain up to 1000+ virtual users; monotonic timing with event-loop lag auto-check. **Round:** 4
- **DEC-010:** Architecture is a modular monolith (layered modules behind interfaces); phases map onto modules. **Rationale:** simple, testable, evolvable; plugin architecture is YAGNI; wrapping an existing tool constrains measurement conventions. **Alternatives:** plugin entry points, extend vLLM/LLMPerf/locust. **Round:** 2 (Phase 2 approach)
- **DEC-011:** Embedding/judge use OpenAI-compatible API; judge default `ibm/claude-haiku-4-5` via `IBM_ICA_*`; embedding local if `url` omitted; config supports `$ENV:` interpolation; embedding `threshold` mandatory. **Round:** 3 (carried from brainstorming)
- **DEC-012:** Goodput with interactive + relaxed SLO profiles; percentile set p50/p90/p95/p99 plus mean/min/max/std (p99.9 conditional, p80 dropped); optional cost from registry pricing. **Round:** 4 (metrics challenge)
